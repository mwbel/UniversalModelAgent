# 可视化规划模块设计

## 目标

把当前“根据提问硬编码推荐可视化”的逻辑，升级成一个可扩展的功能模块：

1. 识别提问中的关键词和概念意图
2. 从“可视化能力注册表”中选择最合适的能力
3. 返回可直接嵌入对话的可视化指令
4. 输出 A2UI-compatible render instruction，并保留 iframe fallback

---

## 参考 `GreanEnderman/edu-viz-core` 的设计点

参考项目里最重要的不是某个页面，而是：

- `capability / manifest` 明确声明组件能力
- Agent 不直接拼前端，而是先选择一个“能力”
- 前端通过统一渲染链路承载
- 组件交互要有 feedback contract，后续可以回流给教学对话

在本项目里，对应关系是：

- `backend/visualization_catalog.py`
  对应 capability registry
- `backend/services/visualization_planner.py`
  对应 Visualizer / capability selector
- `frontend/app.js` 中的内嵌可视化卡片
  对应当前阶段的轻量 renderer

## 当前阶段的定位

这套模块还不是完整的 A2UI 动态生成系统。

它更准确的定位是：

**A2UI-compatible Visualizer 层。**

也就是：

- 先把“问题 -> 能力选择”这层抽干净
- planner 已经输出 A2UI-compatible 指令
- 当前 renderer 先用 iframe / 页面嵌入作为 fallback
- 后续 renderer 可以用同一份 `a2uiInstruction` 替换成真正的 A2UI surface

## 为什么这比硬编码更好

之前的方式：

- 问题 -> if/else -> 返回固定页面

问题：

- 解释不了为什么命中
- 扩一个能力就要改业务逻辑
- 无法平滑升级到 A2UI

现在的方式：

- 问题 -> planner -> capability match -> render instruction

优点：

- 能解释命中原因
- 能单独调试
- 能注册更多能力而不改聊天主流程
- 和 `edu-viz-core` 的 capability 思路一致
