# AhaTutor v2.0 项目介绍 PPT 提纲与内容

> 建议页数：12-14 页  
> 叙事主线：AhaTutor 不是“问答式 AI 家教”，而是围绕真实项目目标组织学习、测评、可视化和产物推进的项目驱动学习系统。

## 1. 封面

**标题**：AhaTutor v2.0  
**副标题**：基于 A2UI 与 PBL 工作流的项目驱动智能学习系统

**页面内容**

- 让学习从“听懂知识”走向“完成作品”
- 项目目标驱动、知识树规划、交互式可视化、掌握度评估
- React + FastAPI + A2UI + 多 Agent 教学编排

**讲述重点**

AhaTutor v2.0 的定位不是传统聊天机器人，而是一个面向学习目标和项目产物的智能学习工作台。用户提出想完成的项目，系统负责拆解路径、组织学习、判断掌握情况，并把知识回收到最终作品中。

**视觉建议**

使用 “The Illuminated Mind” 风格：暖奶油背景、大留白、深靛蓝主标题、金色细线或光晕作为强调。

## 2. 问题背景：传统 AI Tutor 的局限

**页面内容**

- 多数 AI Tutor 停留在“问答”和“解释”层面
- 学习路径通常线性，难以贴合真实目标
- 用户学完知识后，仍然不知道如何转化成作品
- 图文解释缺少可操作、可反馈的交互过程

**讲述重点**

传统 AI 辅导能够解释概念，但很难持续追踪“用户为什么学、学到哪里、下一步应该做什么”。AhaTutor 的出发点是把学习过程绑定到一个真实项目目标上，让知识服务于创造。

**视觉建议**

左侧放传统流程：提问 -> 回答 -> 结束。右侧放 AhaTutor 流程：项目目标 -> 知识树 -> 学习/测评 -> 可视化 -> 项目推进。

## 3. 产品定位：项目驱动学习操作系统

**页面内容**

- 用户不是输入“教我物理”，而是输入“我想做一个弹簧振子模拟器”
- 系统将项目目标转化为结构化学习路径
- 学习节点与项目产物保持关联
- 每个节点都可以进入学习、复习、测评或完成状态

**讲述重点**

系统把用户目标拆成知识节点，而不是给出固定课程目录。比如一个“火星农业温室”项目，会被拆解成植物生长、环境控制、传感器建模、数据分析等节点，学习过程始终围绕项目推进。

**视觉建议**

用中心目标卡片连接到 2-3 层知识树，再连接到右下角的“项目产物”。

## 4. 核心体验流程

**页面内容**

1. 输入项目目标
2. GoalClarifierAgent 解析目标、领域、难度和产物类型
3. CurriculumArchitectAgent 生成知识树
4. 用户选择当前节点掌握情况
5. TutorAgent 进行节点教学
6. AssessmentDesignerAgent / MasteryJudgeAgent 完成测评与判定
7. IntegrationService 将节点学习反馈回项目进展

**讲述重点**

这是当前新前端和后端已经形成的主链路。前端通过欢迎页、规划页、学习页和左右侧栏承载流程；后端通过 PBL API、工作流状态机和教学 Agent 推进流程。

**视觉建议**

做成横向流程图，每一步用一个圆角玻璃卡片；“测评”和“学习”之间可以做成可循环闭环。

## 5. 新前端：从聊天界面升级为学习工作台

**页面内容**

- `AppShell` 管理欢迎、规划、学习、反思等核心状态
- `/projects` 支持项目列表与历史项目入口
- 规划页展示知识树与学习路径
- 学习页整合章节内容、A2UI 可视化、测评、掌握度判定和集成反馈
- 左右侧栏承担导航、进度和上下文辅助

**讲述重点**

新前端不再只是一个聊天窗口，而是一个围绕 PBL 状态变化的学习工作台。用户既可以像聊天一样输入，也可以通过知识树节点、测评卡片、完成按钮和侧栏导航推动学习。

**视觉建议**

放一张三栏布局示意：左侧知识树/章节导航，中间学习内容，右侧状态/进度/操作。

## 6. 新后端：PBL 编排层

**页面内容**

- 新增 `backend/app/routers/pbl.py` 作为 PBL API 入口
- 新增 `learning/` 管理 ProjectGoal、LearningNode、WorkflowState、Assessment 等模型
- 新增 `workflow/` 处理节点状态迁移、下一节点选择和项目里程碑
- 复用原有 Chat、SSE、PromptBuilder、A2UI 能力
- 聊天消息与 PBL 状态分离，支持恢复和追踪

**讲述重点**

后端没有推倒重来，而是在原有对话和 A2UI 能力之上加了一层确定性的学习工作流。LLM 负责生成内容，后端状态机负责流程推进，这样系统更可控、可测试、可恢复。

**视觉建议**

用三层架构图：前端学习工作台 -> PBL 编排 API -> LLM/A2UI/插件/数据库能力。

## 7. 多 Agent 分工

**页面内容**

| Agent / Service | 主要职责 |
| --- | --- |
| GoalClarifierAgent | 将自然语言目标转为结构化 ProjectGoal |
| CurriculumArchitectAgent | 生成 2-3 层知识树与节点依赖 |
| TutorAgent | 围绕当前节点进行教学 |
| VisualizerAgent | 判断是否需要交互式可视化以及使用什么组件 |
| AssessmentDesignerAgent | 为节点生成测评题 |
| MasteryJudgeAgent | 根据答案判断掌握、复习或继续学习 |
| IntegrationService | 把节点学习成果回收到项目目标 |

**讲述重点**

当前系统采用“确定性工作流 + 专职 Agent”的结构。这样既保留 LLM 的生成能力，又避免让单个模型自由决定整个学习流程，降低流程漂移风险。

**视觉建议**

用“教学指挥台”隐喻：Orchestrator 位于中心，周围连接多个专业 Agent。

## 8. A2UI：让 AI 不只会说，还会展示和交互

**页面内容**

- 后端流式输出文本与 A2UI JSONL
- 前端通过 `@a2ui/react` 渲染可交互组件
- PBL 学习页使用 `PblA2UISurface` 承载节点可视化
- 插件系统提供学科组件能力，例如物理模拟、图形工具等
- 用户交互可回传给 AI，形成教学闭环

**讲述重点**

A2UI 是项目的重要差异点。它让 AI 在解释概念时可以同步生成可操作的可视化组件，而不是只输出文字。对于物理、数学、编程等学科，这会明显提升“看见规律、动手验证”的体验。

**视觉建议**

左侧放文本流，右侧放交互组件卡片；中间用 SSE/A2UI JSONL 箭头连接。

## 9. 数据与状态模型

**页面内容**

- `ProjectGoal`：项目标题、领域、产物类型、难度、约束
- `LearningNode`：标题、描述、前置依赖、掌握标准、状态、组件提示、项目关联
- `LearningWorkflowState`：当前节点、当前模式、已掌握/薄弱/跳过节点、项目里程碑
- `Assessment` / `AssessmentAttempt`：测评题、用户答案、得分、薄弱点和建议

**讲述重点**

这些模型让系统从“临时聊天”升级成“可持续学习会话”。刷新页面后可以恢复项目状态；切换节点后能保留对应消息；完成节点后可以生成项目相关反馈。

**视觉建议**

做成四张数据卡片，底部用一条“conversation_id / goal_id / node_id”串联。

## 10. 当前已实现能力

**页面内容**

- 项目目标创建与结构化解析
- 项目列表与项目详情入口
- 知识树生成与规划页展示
- 节点切换、节点动作和工作流状态迁移

**讲述重点**

当前版本已经具备可演示的端到端主链路：输入目标、生成知识树、进入节点学习、触发测评、得到判定、完成节点并收到项目应用反馈。后续重点是提升内容质量、可视化稳定性和插件覆盖度。

**视觉建议**

用“已完成能力矩阵”，行是前端/后端/AI 编排/A2UI，列是能力点。

## 11. 技术架构总览

**页面内容**

- 前端：React、TypeScript、Vite、Tailwind CSS、Framer Motion、A2UI Renderer
- 后端：FastAPI、Pydantic、SQLite、SSE、OpenAI/LLM 服务
- 编排层：PBL Router、LearningOrchestrator、TutorOrchestrator、状态迁移
- 插件层：学科 manifest、组件注册、能力注入、Gallery 预览
- 数据层：会话、消息、目标、节点、工作流、测评和事件

**讲述重点**

架构核心是“前端体验层 + 后端工作流层 + AI 生成层 + 插件能力层”。PBL 编排层是连接所有能力的中枢，A2UI 则是把 AI 生成内容变成互动体验的桥。

**视觉建议**

建议做成洋葱图或分层架构图，突出 PBL 编排层位于中间。

## 12. 创新点

**页面内容**

- 从课程驱动转向项目驱动
- 从单轮问答转向可恢复、可追踪的学习工作流
- 从静态图文解释转向 A2UI 交互式可视化
- 从单一 AI 回答转向多 Agent 专职协作
- 从“学完知识”转向“知识回收到作品”

**讲述重点**

AhaTutor 的创新不只是接入 LLM，而是重新定义学习组织方式：目标在前，知识服务于目标，测评校准路径，可视化辅助理解，最后回到可完成的产物。

**视觉建议**

用“传统 AI Tutor vs AhaTutor v2.0”对比表。

## 13. 风险与下一步

**页面内容**

- 提升知识树质量：加强 schema 校验、结构校验和失败重试
- 提升 A2UI 稳定性：限制组件 schema、增强生成校验和回退策略
- 丰富学科插件：优先扩展物理、数学、编程等高可视化学科
- 完善学习画像：基于节点事件、测评结果和交互记录形成长期记忆
- 强化项目产物桥接：让每个学习节点都能明确推动项目下一步

**讲述重点**

当前系统主链路已跑通，下一阶段不是简单加功能，而是提高“学习路径质量”和“项目推进真实感”。这两点决定 AhaTutor 能否从演示原型走向可长期使用的学习系统。

**视觉建议**

做成路线图：稳定主链路 -> 增强可视化 -> 扩展插件生态 -> 学习画像与作品闭环。

## 14. 结尾页

**标题**：让每一次学习，都通向一个真实作品

**页面内容**

- AhaTutor v2.0 = 项目目标 + 学习路径 + 智能教学 + 交互组件 + 掌握评估 + 产物推进
- 它帮助用户不只是“学会”，而是“做成”

**讲述重点**

用一句话收束：AhaTutor 希望把 AI Tutor 从“解释知识的人”升级为“陪你完成项目的学习伙伴”。

**视觉建议**

中心放一句金色强调文案：“Learn by building. Build by understanding.”

## 可选附录页

### 附录 A：主要接口

- `POST /v1/pbl/goals`：创建并解析项目目标
- `POST /v1/pbl/goals/{goal_id}/tree`：生成知识树
- `GET /v1/pbl/sessions/{conversation_id}/state`：恢复 PBL 状态
- `POST /v1/pbl/sessions/{conversation_id}/switch-node`：切换节点
- `POST /v1/pbl/nodes/{node_id}/action`：处理节点动作
- `POST /v1/pbl/chat`：围绕当前节点进行流式教学
- `POST /v1/pbl/nodes/{node_id}/assessment`：生成测评
- `POST /v1/pbl/assessments/{assessment_id}/submit`：提交答案并判定掌握度

### 附录 B：演示脚本建议

1. 在首页输入一个项目目标，例如“我想做一个火星农业温室模拟器”。
2. 展示系统解析出的项目目标、领域、难度和产物类型。
3. 进入规划页，展示生成的知识树。
4. 选择一个前置节点，点击“不熟悉”进入学习。
5. 展示 TutorAgent 的流式教学内容和 A2UI 可视化区域。
6. 点击“不确定/测评”，展示测评题。
7. 提交答案，展示 MasteryJudgeAgent 的掌握度判定。
8. 完成节点，展示 IntegrationService 如何把知识应用回项目。

### 附录 C：一句话版本

AhaTutor v2.0 是一个项目驱动的智能学习系统：它把用户想完成的真实项目拆解为知识树，通过多 Agent 教学编排、掌握度测评和 A2UI 交互式可视化，帮助用户在完成作品的过程中真正掌握知识。
